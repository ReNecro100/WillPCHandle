## 1. Общие сведения

Полное наименование системы: программная система «Will PC Handle?» для оценки возможности запуска компьютерных игр на персональном компьютере пользователя.

Краткое наименование системы: WillPC.

Основание для разработки: выполнение проекта в рамках учебной практики.

Заказчик: [указать образовательную организацию или подразделение].

Исполнители: [ФИО студента 1], [ФИО студента 2].

Планируемая область применения: персональные компьютеры пользователей, которым необходимо быстро определить, соответствует ли их устройство минимальным или рекомендуемым системным требованиям выбранной игры.

Разрабатываемая система представляет собой настольное приложение с регистрацией пользователей, базой данных, главным экраном со списком игр, экраном подробной информации об игре, профилем пользователя, модулем сканирования характеристик ПК, интеграцией со Steam API и модулем интеллектуального анализа системных требований.

Основная идея системы заключается в том, что пользователь выбирает игру, приложение собирает основные характеристики компьютера, получает сведения об игре и ее требованиях через Steam API, после чего модуль анализа формирует понятный вывод: сможет ли компьютер запустить игру, какие параметры соответствуют требованиям, а какие могут стать ограничением.

Текущая программная реализация проекта выполняется на платформе .NET 8 с использованием Avalonia UI. В проекте уже предусмотрены пользовательские окна главного экрана, страницы игры и пользовательского соглашения.

## 2. Назначение и цели создания системы

### 2.1. Назначение системы

Система предназначена для автоматизации проверки совместимости персонального компьютера пользователя с системными требованиями компьютерных игр.

WillPC должна помогать пользователю до установки или покупки игры определить вероятность ее корректного запуска. Система должна отображать список игр, предоставлять подробную информацию о выбранной игре, сканировать аппаратные и программные характеристики компьютера, сравнивать их с требованиями игры и сохранять результаты проверки в профиле пользователя.

Система должна быть ориентирована на пользователей, которые не обладают глубокими техническими знаниями о характеристиках комплектующих. Поэтому итоговая оценка должна отображаться в простой форме: «потянет», «потянет с ограничениями» или «не потянет», с кратким объяснением причин.

### 2.2. Цели создания системы

Целями создания системы являются:

1. Сокращение времени, необходимого пользователю для ручного поиска и сравнения системных требований игры с характеристиками своего компьютера.

2. Повышение удобства выбора игр за счет автоматической проверки совместимости.

3. Создание единого интерфейса для просмотра игр, их описаний, изображений, системных требований и результата проверки.

4. Реализация учетных записей пользователей и сохранение истории проверок в базе данных.

5. Получение практического опыта разработки настольного приложения с графическим интерфейсом, базой данных, внешним API и модулем интеллектуального анализа.

6. Формирование учебного проекта, демонстрирующего полный цикл разработки: проектирование, реализацию, тестирование и подготовку документации.

## 3. Характеристика объектов автоматизации

Объектом автоматизации является процесс оценки возможности запуска компьютерной игры на персональном компьютере пользователя.

До внедрения системы пользователь обычно выполняет проверку вручную: ищет требования игры на странице магазина или в сторонних источниках, открывает сведения о своем компьютере, сравнивает процессор, видеокарту, объем оперативной памяти, операционную систему и свободное место на диске. Такой процесс занимает время и требует понимания технических характеристик.

Автоматизации подлежат следующие операции:

1. Регистрация и авторизация пользователя.

2. Хранение данных пользователя, списка игр, характеристик ПК и истории проверок.

3. Получение списка игр и сведений о выбранной игре.

4. Обращение к Steam API для получения доступной информации об игре.

5. Сканирование характеристик персонального компьютера пользователя.

6. Извлечение и структурирование системных требований игры.

7. Сравнение характеристик компьютера с минимальными и рекомендуемыми требованиями.

8. Формирование итогового вывода о совместимости.

9. Отображение результата проверки в интерфейсе приложения.

10. Сохранение результатов проверки в профиле пользователя.

Основными участниками процесса являются пользователь приложения, программная система WillPC, база данных, Steam API и модуль интеллектуального анализа.

## 4. Требования к системе

### 4.1. Требования к системе в целом

Система должна быть реализована в виде настольного приложения с графическим интерфейсом.

Приложение должно обеспечивать удобную навигацию между основными экранами: регистрация, авторизация, главный экран со списком игр, экран игры, профиль пользователя.

Интерфейс системы должен быть понятным для пользователя без необходимости изучения дополнительной технической документации. Основные действия должны быть доступны через кнопки, поля ввода, карточки игр и элементы навигации.

Система должна обеспечивать хранение данных в базе данных. В базе данных должны храниться сведения об учетных записях, играх, характеристиках компьютера и результатах проверок. Для учебного проекта допускается использование локальной базы данных SQLite либо другой выбранной СУБД.

Система должна получать данные об играх и системных требованиях через Steam API при наличии доступа к сети Интернет. При недоступности Steam API система должна отображать пользователю сообщение об ошибке и, при наличии локального кэша, использовать сохраненные ранее данные.

Система должна выполнять сканирование характеристик компьютера только после согласия пользователя. Не допускается сбор данных, не связанных с проверкой совместимости игр.

Система должна обеспечивать защиту учетных данных пользователя. Пароли должны храниться в виде хэшей, а не в открытом виде. Ключи доступа к внешним сервисам не должны отображаться в пользовательском интерфейсе.

Система должна корректно обрабатывать ошибки: отсутствие подключения к сети, отсутствие данных по игре, невозможность определить видеокарту или процессор, ошибки базы данных и неверный ввод пользователя.

Система должна быть рассчитана на работу на персональных компьютерах под управлением Windows 10 или Windows 11. При использовании Avalonia UI допускается дальнейшее расширение поддержки на другие операционные системы, если модули сканирования характеристик будут адаптированы под соответствующую платформу.

Ожидаемое время выполнения проверки одной игры при стабильном подключении к сети не должно превышать 10 секунд, без учета задержек внешних сервисов.

### 4.2. Требования к функциям, выполняемым системой

#### 4.2.1. Регистрация пользователя

Система должна предоставлять пользователю возможность создать учетную запись.

При регистрации пользователь должен вводить логин, адрес электронной почты и пароль. Система должна проверять заполненность обязательных полей, корректность формата электронной почты и минимальную длину пароля.

При успешной регистрации данные пользователя должны сохраняться в базе данных. Если логин или адрес электронной почты уже используются, система должна выводить сообщение об ошибке.

#### 4.2.2. Авторизация пользователя

Система должна предоставлять возможность входа в учетную запись по логину или электронной почте и паролю.

При неверных данных система должна выводить понятное сообщение об ошибке. После успешной авторизации пользователь должен переходить на главный экран приложения.

#### 4.2.3. Главный экран со списком игр

Система должна отображать каталог игр в виде карточек. Карточка игры должна содержать название, жанр или краткое описание, изображение и индикатор статуса проверки, если игра уже проверялась пользователем.

На главном экране должна быть предусмотрена возможность перехода к подробной странице игры. Также рекомендуется предусмотреть поиск по названию и фильтрацию по жанру или статусу совместимости.

#### 4.2.4. Экран игры

Система должна отображать подробные сведения о выбранной игре: название, описание, изображения, минимальные и рекомендуемые системные требования, а также кнопку запуска проверки совместимости.

После выполнения проверки на странице игры должен отображаться итоговый результат. Результат должен включать общий статус совместимости, список совпавших требований и список параметров, которые не соответствуют требованиям или могут повлиять на производительность.

#### 4.2.5. Профиль пользователя

Система должна предоставлять экран профиля пользователя.

В профиле должны отображаться основные данные учетной записи, сохраненные характеристики компьютера, история проверок игр и результаты последних проверок.

Пользователь должен иметь возможность выйти из учетной записи. При необходимости может быть предусмотрена возможность повторного сканирования компьютера и обновления сохраненных характеристик.

#### 4.2.6. Сканирование характеристик компьютера

Система должна определять основные характеристики персонального компьютера пользователя:

1. Модель процессора.

2. Количество ядер или потоков процессора, если эти данные доступны.

3. Модель видеокарты.

4. Объем видеопамяти, если эти данные доступны.

5. Объем оперативной памяти.

6. Версию и разрядность операционной системы.

7. Объем свободного места на диске.

8. Версию DirectX или другие важные программные компоненты, если они необходимы для сравнения.

Полученные характеристики должны сохраняться в базе данных как снимок конфигурации. Это позволит повторно использовать результаты сканирования без постоянного обращения к системным данным.

#### 4.2.7. Интеграция со Steam API

Система должна обращаться к Steam API для получения сведений об игре: названия, описания, изображений, жанров, идентификатора игры и системных требований при наличии этих данных.

Полученные данные должны быть преобразованы в формат, удобный для хранения и отображения в приложении. При изменении данных на стороне Steam система должна иметь возможность обновить локальную информацию.

Если API не возвращает системные требования в структурированном виде, система должна передавать текст требований в модуль анализа для дальнейшего извлечения параметров.

#### 4.2.8. Модуль интеллектуального анализа

Модуль интеллектуального анализа должен использоваться для обработки неструктурированных системных требований, полученных из Steam, и формирования понятного объяснения результата.

Модуль должен выделять из текста требований основные параметры: процессор, видеокарту, оперативную память, операционную систему, DirectX и место на диске.

Модуль должен сравнивать требования игры с характеристиками компьютера пользователя. Для процессоров и видеокарт допускается использование справочника производительности, правил сопоставления или внешнего интеллектуального сервиса.

Результат анализа должен включать:

1. Итоговый статус: «потянет», «потянет с ограничениями» или «не потянет».

2. Краткое текстовое объяснение.

3. Таблицу сравнения характеристик ПК и требований игры.

4. Рекомендации, например снижение настроек графики или необходимость обновления комплектующих.

#### 4.2.9. Работа с базой данных

Система должна хранить данные в базе данных. Минимальный состав сущностей:

1. Пользователь.

2. Игра.

3. Системные требования игры.

4. Снимок характеристик компьютера пользователя.

5. Результат проверки совместимости.

6. История действий или проверок пользователя.

База данных должна обеспечивать добавление, чтение, обновление и удаление данных в пределах функций приложения.

#### 4.2.10. Пользовательское соглашение

Система должна отображать пользовательское соглашение или уведомление о том, какие данные собираются при сканировании компьютера и для чего они используются.

Пользователь должен подтвердить согласие перед выполнением сканирования. При отказе от согласия функции сканирования и проверки совместимости должны быть недоступны или ограничены.

### 4.3. Требования к видам обеспечения

#### 4.3.1. Информационное обеспечение

Информационное обеспечение системы должно включать:

1. Данные учетных записей пользователей.

2. Сведения об играх, полученные из Steam API.

3. Минимальные и рекомендуемые системные требования игр.

4. Сведения о характеристиках компьютеров пользователей.

5. Результаты проверок совместимости.

6. Справочные данные для сопоставления моделей процессоров и видеокарт.

Данные должны храниться в структурированном виде и использоваться для отображения в интерфейсе, повторного анализа и формирования истории проверок.

#### 4.3.2. Программное обеспечение

Для разработки системы должны использоваться:

1. Язык программирования C#.

2. Платформа .NET 8.

3. Фреймворк Avalonia UI для создания графического интерфейса.

4. СУБД SQLite или другая выбранная СУБД для хранения данных.

5. Steam API для получения сведений об играх.

6. Внешний или локальный модуль искусственного интеллекта для анализа требований и формирования объяснений.

7. Среда разработки JetBrains Rider или Visual Studio.

#### 4.3.3. Техническое обеспечение

Для работы системы требуется персональный компьютер со следующими минимальными характеристиками:

1. Операционная система Windows 10 или Windows 11.

2. Процессор с поддержкой запуска приложений .NET 8.

3. Оперативная память не менее 4 ГБ.

4. Свободное место на диске не менее 300 МБ для приложения, базы данных и кэша изображений.

5. Подключение к сети Интернет для обращения к Steam API и внешнему модулю искусственного интеллекта.

#### 4.3.4. Лингвистическое обеспечение

Пользовательский интерфейс системы должен быть выполнен на русском языке. Названия игр, жанры и описания могут отображаться на языке, предоставленном Steam API.

Сообщения об ошибках, статусы проверки и рекомендации должны быть понятными для пользователя и не должны содержать избыточных технических терминов без пояснения.

#### 4.3.5. Математическое и алгоритмическое обеспечение

Алгоритм проверки совместимости должен включать следующие этапы:

1. Получение характеристик компьютера пользователя.

2. Получение системных требований игры.

3. Нормализация текстовых данных и выделение ключевых параметров.

4. Сравнение объема оперативной памяти, свободного места на диске, версии операционной системы и DirectX.

5. Сопоставление процессора и видеокарты с требованиями игры по справочнику или правилам производительности.

6. Расчет итогового статуса совместимости.

7. Формирование пояснения результата.

Для каждого требования должен определяться один из статусов: соответствует, частично соответствует, не соответствует, невозможно определить.

#### 4.3.6. Организационное обеспечение

Разработка системы должна выполняться исполнителями проекта в рамках учебной практики. Распределение обязанностей между участниками проекта определяется командой.

Рекомендуемое распределение работ:

1. Проектирование интерфейса и реализация экранов приложения.

2. Реализация базы данных и учетных записей пользователей.

3. Реализация модуля сканирования характеристик ПК.

4. Реализация интеграции со Steam API.

5. Реализация модуля анализа и формирования результата.

6. Тестирование и подготовка документации.

## 6. Порядок контроля и приемки системы

Контроль разработки должен выполняться на каждом этапе создания системы. Основными формами контроля являются просмотр исходного кода, проверка интерфейса, функциональное тестирование и демонстрация работы приложения руководителю практики.

Приемка системы выполняется после завершения основных функций.

К приемке должны быть представлены:

1. Рабочее приложение.

2. Исходный код проекта.

3. База данных или скрипт ее создания.

4. Техническое задание.

5. Руководство пользователя.

6. Материалы для демонстрации проекта.

Система считается принятой, если выполнены следующие условия:

1. Пользователь может зарегистрироваться и войти в систему.

2. Главный экран отображает список игр.

3. Пользователь может открыть страницу выбранной игры.

4. Система может получить или использовать данные об игре и ее требованиях.

5. Система может определить основные характеристики ПК пользователя.

6. Система формирует результат проверки совместимости.

7. Результат проверки сохраняется и отображается в профиле пользователя.

8. Ошибки сети, API, базы данных и неверного ввода обрабатываются без аварийного завершения приложения.

9. Интерфейс приложения соответствует основным требованиям удобства использования.

Для проверки системы должны быть подготовлены тестовые сценарии:

1. Регистрация нового пользователя.

2. Попытка регистрации с уже существующим логином.

3. Авторизация с корректными и некорректными данными.

4. Просмотр списка игр.

5. Открытие страницы игры.

6. Сканирование характеристик ПК.

7. Проверка игры, требования которой ниже характеристик ПК.

8. Проверка игры, требования которой выше характеристик ПК.

9. Проверка поведения системы при отсутствии подключения к сети.

10. Просмотр истории проверок в профиле.

## 7. Требования к составу и содержанию работ по подготовке объекта автоматизации к вводу системы в действие

Перед вводом системы в действие необходимо выполнить подготовительные работы.

На компьютере пользователя или демонстрационном компьютере должна быть установлена среда выполнения .NET, если приложение не поставляется как самостоятельная сборка.

Должна быть создана и настроена база данных. В базе данных должны присутствовать необходимые таблицы, а также тестовые данные для демонстрации работы приложения.

Должен быть настроен доступ к Steam API. При использовании ключей или токенов доступа они должны храниться отдельно от пользовательского интерфейса и не должны попадать в публичную часть проекта.

Если для интеллектуального анализа используется внешний сервис, должен быть настроен доступ к нему. При отсутствии доступа к внешнему сервису должна быть предусмотрена демонстрационная или локальная логика сравнения.

Перед демонстрацией необходимо проверить подключение к сети Интернет, наличие тестовой учетной записи, корректность отображения изображений игр и возможность выполнения проверки совместимости.

Пользователь должен быть ознакомлен с тем, какие характеристики компьютера считывает приложение и для чего они используются.

## 8. Требования к документированию

В составе проекта должна быть подготовлена следующая документация:

1. Техническое задание на разработку системы.

2. Описание программы.

3. Руководство пользователя.

4. Описание структуры базы данных.

5. Программа и методика испытаний или список тестовых сценариев.

6. Пояснительная записка к учебному проекту, если она требуется руководителем практики.

Документация должна быть оформлена в едином стиле. Текст должен быть написан на русском языке, с использованием делового стиля изложения. Заголовки разделов должны соответствовать структуре технического задания.

Руководство пользователя должно описывать запуск приложения, регистрацию, вход, выбор игры, выполнение проверки совместимости и просмотр результата в профиле.

Описание программы должно включать назначение системы, используемые технологии, структуру модулей и порядок взаимодействия с базой данных и внешними сервисами.

## 9. Источники разработки

При разработке технического задания использованы следующие источники:

1. Задание на учебную практику по разработке программной системы «Will PC Handle?».

2. Материалы проекта WillPC: структура приложения, графический интерфейс, файлы проекта .NET и Avalonia UI.

3. Требования к структуре технического задания, предоставленные руководителем практики.

4. ГОСТ 34.602-89 «Техническое задание на создание автоматизированной системы».

5. Документация платформы .NET 8.

6. Документация Avalonia UI.

7. Документация Steam Web API и открытые данные магазина Steam.

8. Общие требования к оформлению учебной проектной документации.
